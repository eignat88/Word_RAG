# Функциональный дизайн: База знаний по Word-документам (RAG)

## 1. Назначение
Система должна:
- выполнять семантический поиск по содержимому `.docx` документов;
- отвечать на вопросы пользователя в формате чат-ассистента;
- генерировать ответы строго на основе загруженных документов.

## 2. Цели
Поддержать сценарии:
- быстрый поиск информации по ФД (например, `DAX-7407`);
- анализ алгоритмов, интерфейсов и процессов;
- ответы на вопросы вида:
  - «где используется `WMS_IsOversizedItemIM`?»
  - «как работает алгоритм закрытия ячеек?»

## 3. Входные данные
- Формат: `.docx`.
- Типичная структура разделов:
  - Назначение
  - Алгоритм
  - Интерфейс
  - Прочие разделы

## 4. Выходные данные
### 4.1 Поиск
Список релевантных фрагментов:
- текст фрагмента;
- документ (ФД);
- раздел.

### 4.2 Ответ
- сгенерированный ответ на вопрос;
- ответ основан на найденных фрагментах;
- отображение источников.

## 5. Общая логика (RAG)
1. Загрузка документов.
2. Разбиение текста на чанки.
3. Генерация embedding для каждого чанка.
4. Сохранение в PostgreSQL + pgvector.
5. По запросу пользователя:
   - embedding запроса;
   - поиск похожих чанков;
   - передача контекста в LLM;
   - генерация ответа.

## 6. Архитектура
Компоненты:
- парсер документов (Python);
- генерация embeddings (локально через Ollama);
- PostgreSQL + pgvector;
- сервис поиска;
- сервис генерации ответов;
- UI (чат, опционально).

## 7. Структура БД
Таблица `knowledge_base`:

| Поле | Тип | Описание |
|---|---|---|
| id | int | Идентификатор |
| document_name | text | Название документа |
| fd_number | text | Номер ФД (`DAX-****`) |
| section | text | Раздел (Алгоритм, Интерфейс и т.д.) |
| chunk_text | text | Текст фрагмента |
| embedding | vector | Вектор представления |

## 8. Алгоритмы
### 8.1 Загрузка документов
- Пользователь указывает папку с `.docx`.
- Система читает документы, извлекает текст, определяет разделы по заголовкам.

### 8.2 Чанкинг
Требования:
- размер: `300–1000` символов;
- один раздел должен маппиться в один/несколько чанков;
- запрещено смешивать разные разделы в одном чанке.

### 8.3 Embeddings
Для каждого чанка:
- вызов локальной embedding-модели в Ollama;
- сохранение в БД.

### 8.4 Поиск
Вход: текст вопроса.

Шаги:
1. embedding вопроса;
2. SQL-поиск по косинусной/евклидовой близости (`ORDER BY embedding <-> query_embedding`);
3. `LIMIT N` (например, 5);
4. возврат топ-N чанков.

### 8.5 Генерация ответа
Вход:
- вопрос;
- найденные чанки.

Промпт-шаблон:

```text
Контекст:
<чанки>

Вопрос:
<вопрос>

Ответь строго на основе контекста.
```

## 9. Функциональные требования
- загрузка документов из папки;
- повторная загрузка (переиндексация/обновление);
- поиск по тексту вопроса;
- вывод топ-N результатов;
- фильтрация по `fd_number` и `section`;
- чат-интерфейс с источниками.

## 10. Нефункциональные требования
- полностью локальная работа (без внешних API);
- время ответа:
  - поиск: `< 1 сек`;
  - ответ: `< 5–10 сек`;
- поддержка `1000+` документов.

## 11. Ограничения
- ответы формируются только по загруженным документам;
- качество зависит от структуры документов и чанкинга;
- требуется предварительная индексация.

## 12. Тестовые сценарии
### Тест 1
Вопрос: «где используется `WMS_IsOversizedItemIM`?»

Ожидание:
- найдены ФД `DAX-7407` / `DAX-11367`;
- ответ содержит ссылки на документы.

### Тест 2
Вопрос: «алгоритм закрытия ячеек»

Ожидание:
- найден раздел «Алгоритм»;
- сформирован связный ответ.

### Тест 3
Фильтр: `fd_number = DAX-7407`

Ожидание:
- поиск только внутри указанного документа.

## 13. Этапы реализации
1. Подготовка БД (PostgreSQL + pgvector).
2. Разработка парсера Word.
3. Реализация чанкинга.
4. Интеграция embeddings (Ollama).
5. Реализация поиска.
6. Интеграция LLM.
7. UI (чат).

## 14. Риски
- плохие чанки → плохой поиск;
- длинные документы → медленная индексация;
- слабая структура исходных ФД.

## 15. Рекомендации
- сохранять структуру документа (разделы);
- экспериментировать со стратегиями чанкинга;
- логировать запросы пользователей;
- добавить кэширование.
